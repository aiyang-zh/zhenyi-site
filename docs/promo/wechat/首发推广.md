## 正文（可直接分段粘贴）

**【开篇】**  
做 IM、游戏服、推送这类业务，后台往往绕不开长连接、消息怎么进业务、多机之后怎么协作。我们把这几年沉淀的一套方案开源了，名字叫 **zhenyi**：希望把「能跑通的路径」和「能对照的文档」一起放在 GitHub 上。

**【zhenyi 是什么】**  
简单说，它是一套**面向实时业务的 Go 后端方案**。底层高性能组件放在 **zhenyi-base**（MIT）；在上层提供 **Actor 运行时、统一网关（支持 TCP/WebSocket/KCP）、服务发现、跨进程消息、指标与追踪** 等，让你少重复造轮子，多把时间花在业务上。

**【适合谁】**  
已经在写 Go、准备或正在做**长连接、低延迟、消息驱动**后台的开发者。若你只想找传统 CRUD 脚手架，它可能不是最轻的那类；若你关心网关、Actor、分布式协作怎么落在**同一条仓库叙事**里，值得打开看看。

**【怎么体验】**  
仓库里带了单机 IM 示例，**几分钟可跑通**，且单机示例**不依赖** Etcd/NATS。克隆主仓库后按文档执行 `im_single_demo` 与 `im_single_client` 即可。具体命令见 GitHub 上的 **EXAMPLES** 说明。

**【图书】**  
我们同时维护了一份与代码目录对齐的开源书稿，官网提供在线阅读，方便从第一章跟到附录。书稿与实现不一致时，以源码为准。

**【协议说明】**  
开源遵循 **AGPL-3.0**，商业与闭源场景请先阅读仓库中的 **商业授权说明**。使用第三方依赖时，也请遵守各自许可证。

**【结尾】**  
欢迎转发给同做实时后台的同事。Issue、PR、文档纠错都欢迎；若觉得有用，也可到 GitHub 给仓库一个 Star，方便我们持续维护。
