## 正文（可直接分段粘贴）

做 IM、游戏服、推送这类业务，后台往往绕不开几件事：长连接怎么接、消息怎么进业务、多机之后怎么协作、线上出了问题怎么看。这篇文章，想把我们这几年沉淀的一套方案摊开来讲清楚——它叫 **zhenyi**，是一套用 Go 写的实时后端骨架。

**zhenyi 是什么？**

简单说，它是一套**面向实时业务的 Go 后端方案**。底层高性能组件放在 **zhenyi-base**（MIT 授权），比如 TCP/WebSocket 实现、无锁队列 zqueue 等；在这个基础上，zhenyi 提供：

- **Actor 运行时**：服务级 Actor、MPSC 邮箱、Tick/RPC 等，让你用消息驱动去组织业务，而不是到处散落锁和 channel。  
- **统一网关 zgate**：支持 TCP / WebSocket / KCP，把接入层、会话管理、路由和 Actor 接在一起，避免「网络层一套、业务层又一套」。  
- **分布式能力**：集成 Etcd 做服务发现、NATS 做跨进程总线，加上路由策略（包括 RendezvousHash），可以从单进程自然走向多进程、多节点。  
- **可观测性**：指标、健康探针、链路追踪等，可以和现有的 Prometheus 等体系接上，而不是只能靠日志猜。  

它不是一个通用 Web 框架，而是偏向 **实时、长连接、消息驱动** 的后端骨架。

**适合谁？**

- 已经在日常用 Go，准备或正在做游戏后台、IM、推送、IoT 等**长连接、低延迟**业务。  
- 希望用 Actor 把业务拆成清晰的服务单元，而不是一团 goroutine 加锁的同学。  
- 希望有一条从「网络接入 → Actor 执行 → 多机协作 → 可观测」**连起来的路径**，而不是自己在不同仓库之间拼图。

如果你只需要一个最轻量的 CRUD 脚手架，zhenyi 可能不是最合适的选择；但如果你关心的是「实时后台应该长什么样」，它值得你打开代码看一看。

**怎么快速体验？**

我们在仓库里准备了一个**单机 IM 示例**，几分钟就能跑通一条闭环，而且不依赖 Etcd 和 NATS：

1. 克隆并进入项目：

   ```bash
   git clone https://github.com/aiyang-zh/zhenyi.git
   cd zhenyi
   ```

2. 启动服务端（Gate + IM Actor）：

   ```bash
   go run ./examples/im_single_demo
   ```

3. 另开一个终端，启动客户端，连接本地 8001 端口，随便起一个用户 ID 和昵称：

   ```bash
   go run ./examples/im_single_client \n     -addr 127.0.0.1:8001 \n     -user 10001 \n     -nick alice \n     -room lobby \n     -msgLogin 1 \n     -msgJoin 2 \n     -msgLeave 3 \n     -msgSend 4
   ```

终端里能看到登录、进房、发消息的完整路径；想看细节，可以对照仓库里的示例和文档继续往下钻。

如果你关心的是多进程、服务发现和跨进程路由，可以再看 `im_multi_demo` 等示例，配好 Etcd 和 NATS 之后体验分布式版本。

**配套书稿与在线阅读**

为了让实现和叙述更容易对上号，我们写了一份配套书稿 **《Go Actor 模型：实时应用架构与实战》**，和仓库目录结构对齐，从网络层、Actor 运行时、网关，到分布式和可观测，一路配着 IM 示例讲。

- 书稿源码在 zhenyi 仓库的 `docs/books/go-actor-realtime/` 下。  
- 官网提供了静态构建版，带目录和搜索，方便直接在线阅读。

代码和书稿不是概念上的两套东西，而是可以互相对照的两条视角：你可以先跑示例，再看书里的那一章是怎么讲的；也可以先阅读，再在仓库里翻到对应包的实现。

**最后**

我们希望 zhenyi 能成为一套「**看得懂、跑得动、改得动**」的实时后台方案，而不是一团难以拆解的黑盒。如果你在阅读或使用过程中有任何疑问、发现 bug 或者有改进建议，都欢迎通过 Issue、PR 的方式告诉我们。

如果这篇文章对你有帮助，也欢迎转发给同样在做实时后台的同事朋友，或者到 GitHub 给仓库点一个 Star，这会是我们继续完善这套方案的动力。
