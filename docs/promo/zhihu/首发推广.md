**摘要（约150字）**：做游戏服、IM、推送等长连接后台时，常同时面临连接接入、消息驱动建模、多机协作与可观测。zhenyi 提供一套 Go Actor 实时方案：zhenyi-base（MIT）承接底层组件，上层提供统一网关 zgate、服务发现与跨进程总线，并以示例把路径走通。

## 一、先回答一个问题：这是给谁用的？

如果你在做**游戏服、IM、推送、设备长连接**这类后台，大概率会同时遇到几件事：连接怎么接、消息怎么进业务、多协程下状态怎么不打架、多机之后怎么发现和转发。zhenyi 的定位，就是把这些能力**收拢到一套可运行的 Go 方案里**：底层性能与基础组件放在 **zhenyi-base（MIT）**，框架与网关、分布式协作放在 **zhenyi（AGPL + 商业双授权）**，用示例把路径走通。

它不是「又一个玩具 Web 框架」，而是偏**实时、长连接、消息驱动**的后端骨架。

## 二、开源里具体有什么？

概括成几条，便于你对照自己的需求：

1. **Actor 运行时**：服务级 Actor、MPSC 邮箱、Tick/RPC 等，把业务拆成可消息驱动的单元，而不是到处散落同步原语。  
2. **统一网关 `zgate`**：TCP / WebSocket / KCP 等接入，和会话、路由衔接，减少网络层与业务层重复造轮子。  
3. **分布式相关**：Etcd 服务发现、NATS 跨进程、远程路由（含 RendezvousHash 等），示例里有多进程 IM 可参考。  
4. **可观测**：指标、健康探针、追踪等可接入，方便从「能跑」往「能运维」走。

更细的模块与边界，建议直接看仓库 **[文档索引](https://github.com/aiyang-zh/zhenyi/blob/main/docs/DOCS_INDEX.md)**，避免本文与版本漂移不一致。

## 三、为什么现在开源？

把代码和文档放在公开仓库，是为了让**实现、示例与叙述**能被一起审阅、勘误和演进。首发版本对应说明见 **[v0.1.0 发布说明](https://github.com/aiyang-zh/zhenyi/blob/main/docs/RELEASE_NOTES_v0.md)**（若后续有小版本，以仓库 `RELEASE_NOTES` 为准）。

需要强调：**开源不等于「已替你完成生产验证」**。是否用于关键业务，仍要结合你的压测、灰度与合规要求；仓库 README 与书稿里也保留了相应免责表述。

## 四、如何快速验证「适不适合我」？

若你关心的是「能不能在本地几命令跑通一条闭环」，可以用单机 IM 示例（**无需 Etcd/NATS**）：

```bash
git clone https://github.com/aiyang-zh/zhenyi.git
cd zhenyi
go run ./examples/im_single_demo
```

另开终端运行 `im_single_client`，按仓库 **[EXAMPLES](https://github.com/aiyang-zh/zhenyi/blob/main/docs/EXAMPLES.md)** 参数说明即可互动。  
若你更关心分布式与发现，再跟多进程示例与 Etcd/NATS 依赖。
